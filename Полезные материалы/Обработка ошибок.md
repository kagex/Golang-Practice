# Обработка ошибок в Go

## Введение

В Go ошибки — это **значения**, а не исключения. Функции возвращают ошибку как обычное возвращаемое значение типа `error`, а вызывающий код обязан её проверить. Это делает обработку ошибок явной и предсказуемой.

> 💡 **Философия Go:** "Errors are values" (Роб Пайк). Ошибки не прерывают поток выполнения — их нужно обрабатывать явно.

---

## Интерфейс `error`

`error` — это встроенный интерфейс с единственным методом:

```go
type error interface {
    Error() string
}
```

Любой тип, реализующий метод `Error() string`, считается ошибкой.

### Нулевое значение

Нулевое значение `error` — это `nil`. Если функция возвращает `nil` вместо ошибки, значит операция выполнена успешно.

```go
var err error // err == nil
fmt.Println(err) // <nil>
```

## Создание ошибок

### 1. `errors.New()` — простая ошибка

Создаёт ошибку с фиксированным текстом.

```go
import "errors"

err := errors.New("что-то пошло не так")
fmt.Println(err) // что-то пошло не так
```

### 2. `fmt.Errorf()` — форматированная ошибка

Создаёт ошибку с форматированным сообщением (аналог `fmt.Sprintf`, но возвращает `error`).

```go
import "fmt"

name := "Ivan"
err := fmt.Errorf("пользователь %q не найден", name)
fmt.Println(err) // пользователь "Ivan" не найден
```

### 3. Кастомный тип ошибки

Создайте свой тип, реализующий интерфейс `error`:

```go
type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("ошибка валидации поля %q: %s", e.Field, e.Message)
}

// Использование
err := &ValidationError{
    Field:   "email",
    Message: "неверный формат",
}
fmt.Println(err) // ошибка валидации поля "email": неверный формат
```

**Преимущества кастомных типов:**
- Можно хранить дополнительные данные (коды, контекст)
- Можно проверять тип через `errors.As()`
- Можно реализовать дополнительные методы

## Возврат ошибок из функций

Стандартный паттерн — возвращать ошибку **последним** значением:

```go
func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, errors.New("деление на ноль")
    }
    return a / b, nil
}

// Использование
result, err := divide(10, 0)
if err != nil {
    fmt.Println("Ошибка:", err)
    return
}
fmt.Println("Результат:", result)
```

### Go-паттерн: `if err != nil`

Это самая частая конструкция в Go-коде:

```go
if err != nil {
    return err // или обработка ошибки
}
```

### Early return

Рекомендуется обрабатывать ошибки сразу и возвращаться из функции:

```go
func process(data []byte) error {
    if len(data) == 0 {
        return errors.New("пустые данные")
    }
    
    if err := validate(data); err != nil {
        return fmt.Errorf("валидация: %w", err) // оборачивание
    }
    
    // основная логика
    return nil
}
```

## Оборачивание ошибок

Оборачивание позволяет добавить контекст к ошибке, сохранив исходную ошибку в цепочке.

### Оператор `%w` в `fmt.Errorf`

```go
func readFile(path string) ([]byte, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, fmt.Errorf("чтение файла %q: %w", path, err)
    }
    return data, nil
}
```

Теперь ошибка содержит:
- Контекст: `"чтение файла \"config.json\": "`
- Исходную ошибку: `open config.json: no such file or directory`

### ⚠️ Важно: `%w` vs `%v`

| Оператор | Поведение |
|----------|-----------|
| `%w` | Оборачивает ошибку — можно извлечь через `errors.Is/As` |
| `%v` | Просто форматирует как строку — исходная ошибка теряется |

```go
err := errors.New("original error")

wrapped1 := fmt.Errorf("context: %w", err) // ✅ можно извлечь
wrapped2 := fmt.Errorf("context: %v", err) // ❌ ошибка потеряна
```

### Оборачивание нескольких ошибок

Начиная с Go 1.20 можно оборачивать несколько ошибок:

```go
err1 := errors.New("error 1")
err2 := errors.New("error 2")

combined := fmt.Errorf("multiple errors: %w, %w", err1, err2)

// errors.Is найдёт любую из обёрнутых ошибок
errors.Is(combined, err1) // true
errors.Is(combined, err2) // true
```

## Проверка ошибок

### `errors.Is()` — проверка на конкретную ошибку

Проверяет, содержится ли в цепочке ошибок конкретная ошибка-значение (sentinel error):

```go
import (
    "errors"
    "os"
)

func readFile(path string) error {
    _, err := os.ReadFile(path)
    if err != nil {
        if errors.Is(err, os.ErrNotExist) {
            return fmt.Errorf("файл не найден: %w", err)
        }
        if errors.Is(err, os.ErrPermission) {
            return fmt.Errorf("нет доступа: %w", err)
        }
        return err
    }
    return nil
}
```

**Почему `errors.Is`, а не `==`?**
Потому что ошибка может быть обёрнута:

```go
err := os.ErrNotExist
wrapped := fmt.Errorf("context: %w", err)

wrapped == os.ErrNotExist          // ❌ false (разные объекты)
errors.Is(wrapped, os.ErrNotExist) // ✅ true (проверяет цепочку)
```

### `errors.As()` — извлечение ошибки нужного типа

Извлекает из цепочки первую ошибку указанного типа:

```go
import (
    "errors"
    "os"
)

func processFile(path string) error {
    _, err := os.Open(path)
    if err != nil {
        var pathErr *os.PathError
        if errors.As(err, &pathErr) {
            // pathErr теперь содержит извлечённую ошибку
            return fmt.Errorf("ошибка пути %s: %w", pathErr.Path, err)
        }
        return err
    }
    return nil
}
```

**Важно:** В `errors.As` нужно передавать **указатель** на переменную нужного типа:

```go
var target *MyError
if errors.As(err, &target) { // ✅ правильно
    // target содержит извлечённую ошибку
}
```

## Sentinel errors (ошибки-переменные)

Это предопределённые переменные-ошибки для сравнения через `errors.Is`:

```go
package mypackage

import "errors"

// Sentinel errors
var (
    ErrNotFound     = errors.New("not found")
    ErrUnauthorized = errors.New("unauthorized")
    ErrInvalidInput = errors.New("invalid input")
)

func GetUser(id int) (*User, error) {
    if id < 0 {
        return nil, ErrInvalidInput
    }
    // ...
    return nil, ErrNotFound
}
```

**Использование:**

```go
user, err := GetUser(-1)
if err != nil {
    switch {
    case errors.Is(err, mypackage.ErrNotFound):
        fmt.Println("Пользователь не найден")
    case errors.Is(err, mypackage.ErrUnauthorized):
        fmt.Println("Нет доступа")
    case errors.Is(err, mypackage.ErrInvalidInput):
        fmt.Println("Неверные данные")
    default:
        fmt.Println("Неизвестная ошибка:", err)
    }
}
```

### ⚠️ Правила sentinel errors

1. Имена должны начинаться с `Err` или `err`
2. Объявляйте их через `var`, а не `const` (т.к. `error` — это интерфейс)
3. Экспортируйте (заглавная буква), если они нужны другим пакетам
4. Используйте `errors.Is` для проверки, а не `==`

## Стандартные ошибки Go

### Пакет `errors`

| Ошибка | Описание |
|--------|----------|
| `errors.ErrUnsupported` | Операция не поддерживается (Go 1.21+) |

### Пакет `os`

| Ошибка | Описание |
|--------|----------|
| `os.ErrNotExist` | Файл/директория не существует |
| `os.ErrExist` | Файл/директория уже существует |
| `os.ErrPermission` | Нет прав доступа |
| `os.ErrClosed` | Ресурс уже закрыт |
| `os.ErrInvalid` | Неверный аргумент |
| `os.ErrDeadlineExceeded` | Превышен таймаут |

### Пакет `io`

| Ошибка | Описание |
|--------|----------|
| `io.EOF` | Достигнут конец файла |
| `io.ErrClosedPipe` | Труба закрыта |
| `io.ErrNoProgress` | Чтение не продвигается |
| `io.ErrShortBuffer` | Буфер слишком мал |
| `io.ErrShortWrite` | Запись неполная |
| `io.ErrUnexpectedEOF` | Неожиданный конец файла |

### Пакет `context`

| Ошибка | Описание |
|--------|----------|
| `context.Canceled` | Контекст отменён |
| `context.DeadlineExceeded` | Превышен дедлайн |

### Пример использования

```go
file, err := os.Open("config.json")
if err != nil {
    if errors.Is(err, os.ErrNotExist) {
        fmt.Println("Файл не существует, используем настройки по умолчанию")
        return defaultConfig, nil
    }
    if errors.Is(err, os.ErrPermission) {
        return nil, fmt.Errorf("нет прав на чтение config.json: %w", err)
    }
    return nil, fmt.Errorf("ошибка открытия файла: %w", err)
}
defer file.Close()
```

## Panic и Recover

### Когда использовать `panic`

`panic` — это **исключение**, которое прерывает выполнение программы. Используйте его **только** для фатальных ошибок, которые нельзя обработать:

✅ **Подходит для panic:**
- Невозможное состояние (инварианты нарушены)
- Ошибки инициализации при старте программы
- Ошибки программиста (неверные аргументы в библиотеке)

❌ **НЕ подходит для panic:**
- Ошибки ввода-вывода
- Ошибки валидации
- Ошибки сети
- Любые ожидаемые ошибки

### Примеры panic

```go
// 1. Невозможное состояние
func mustParse(s string) int {
    n, err := strconv.Atoi(s)
    if err != nil {
        panic(fmt.Sprintf("невозможно преобразовать %q в int: %v", s, err))
    }
    return n
}

// 2. Ошибка инициализации
var globalConfig = mustLoadConfig() // если не загрузится — паника

// 3. Ошибка программиста
func SetTimeout(d time.Duration) {
    if d < 0 {
        panic("timeout cannot be negative")
    }
    // ...
}
```

### Recover — перехват panic

`recover` позволяет перехватить panic и продолжить выполнение:

```go
func safeDivide(a, b int) (result int, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("паника: %v", r)
        }
    }()
    
    if b == 0 {
        panic("деление на ноль")
    }
    return a / b, nil
}

// Использование
result, err := safeDivide(10, 0)
if err != nil {
    fmt.Println("Ошибка:", err) // паника: деление на ноль
} else {
    fmt.Println("Результат:", result)
}
```

### ⚠️ Правила работы с recover

1. `recover` работает **только внутри `defer`**
2. `recover` перехватывает panic только из **текущей горутины**
3. После `recover` функция возвращает нулевые значения
4. Не злоупотребляйте — лучше обрабатывать ошибки явно

### Panic в горутинах

Panic в горутине **убивает всю программу**, если не перехвачен:

```go
// ❌ Плохо — паника убьёт программу
go func() {
    panic("что-то сломалось")
}()

// ✅ Хорошо — перехватываем panic
go func() {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("panic recovered: %v", r)
        }
    }()
    // код, который может паниковать
}()
```

## Лучшие практики

### 1. Не игнорируйте ошибки

```go
// ❌ Плохо
data, _ := os.ReadFile("config.json")

// ✅ Хорошо
data, err := os.ReadFile("config.json")
if err != nil {
    return fmt.Errorf("чтение конфига: %w", err)
}
```

### 2. Оборачивайте ошибки с контекстом

```go
// ❌ Плохо — непонятно, где произошла ошибка
return err

// ✅ Хорошо — понятна цепочка вызовов
return fmt.Errorf("обработка пользователя %d: %w", userID, err)
```

### 3. Обрабатывайте ошибку только один раз

```go
// ❌ Плохо — логируем ошибку несколько раз
func process() error {
    if err := doSomething(); err != nil {
        log.Printf("ошибка: %v", err) // лог #1
        return fmt.Errorf("doSomething: %w", err)
    }
    return nil
}

func main() {
    if err := process(); err != nil {
        log.Printf("ошибка: %v", err) // лог #2 (дублирование!)
    }
}

// ✅ Хорошо — логируем только в одном месте
func process() error {
    if err := doSomething(); err != nil {
        return fmt.Errorf("doSomething: %w", err)
    }
    return nil
}

func main() {
    if err := process(); err != nil {
        log.Printf("ошибка в process: %v", err) // единственный лог
    }
}
```

### 4. Не раскрывайте внутренности

```go
// ❌ Плохо — раскрываем детали реализации
func GetUser(id int) (*User, error) {
    row := db.QueryRow("SELECT * FROM users WHERE id = ?", id)
    if err := row.Scan(&user); err != nil {
        return nil, fmt.Errorf("SQL error: %v", err) // ❌
    }
    return user, nil
}

// ✅ Хорошо — скрываем детали
func GetUser(id int) (*User, error) {
    row := db.QueryRow("SELECT * FROM users WHERE id = ?", id)
    if err := row.Scan(&user); err != nil {
        if errors.Is(err, sql.ErrNoRows) {
            return nil, ErrNotFound
        }
        return nil, fmt.Errorf("получение пользователя: %w", err)
    }
    return user, nil
}
```

### 5. Используйте sentinel errors для известных состояний

```go
var (
    ErrNotFound     = errors.New("not found")
    ErrUnauthorized = errors.New("unauthorized")
)

func GetUser(id int) (*User, error) {
    // ...
    if notFound {
        return nil, ErrNotFound
    }
    return user, nil
}
```

### 6. Используйте кастомные типы для сложных ошибок

```go
type APIError struct {
    StatusCode int
    Message    string
    Details    map[string]string
}

func (e *APIError) Error() string {
    return fmt.Sprintf("API error %d: %s", e.StatusCode, e.Message)
}

// Использование
err := &APIError{
    StatusCode: 404,
    Message:    "user not found",
}

var apiErr *APIError
if errors.As(err, &apiErr) {
    fmt.Printf("Status: %d, Message: %s\n", apiErr.StatusCode, apiErr.Message)
}
```

## Таблица сравнения: когда что использовать

| Ситуация | Что использовать | Пример |
|----------|------------------|--------|
| Простая ошибка | `errors.New()` | `errors.New("invalid input")` |
| Ошибка с контекстом | `fmt.Errorf()` | `fmt.Errorf("user %d: %w", id, err)` |
| Известное состояние | Sentinel error | `ErrNotFound`, `io.EOF` |
| Сложная ошибка с данными | Кастомный тип | `&ValidationError{Field: "email"}` |
| Проверка типа ошибки | `errors.As()` | `errors.As(err, &pathErr)` |
| Проверка конкретной ошибки | `errors.Is()` | `errors.Is(err, os.ErrNotExist)` |
| Фатальная ошибка | `panic` | `panic("impossible state")` |
| Перехват panic | `recover` в `defer` | `if r := recover(); r != nil` |

## Частые ошибки

### 1. Использование `==` вместо `errors.Is`

```go
// ❌ Плохо
if err == os.ErrNotExist {
    // не сработает, если ошибка обёрнута
}

// ✅ Хорошо
if errors.Is(err, os.ErrNotExist) {
    // сработает даже для обёрнутых ошибок
}
```

### 2. Потеря исходной ошибки

```go
// ❌ Плохо — исходная ошибка потеряна
return fmt.Errorf("context: %v", err)

// ✅ Хорошо — исходная ошибка сохранена
return fmt.Errorf("context: %w", err)
```

### 3. Panic для обычных ошибок

```go
// ❌ Плохо
func ReadFile(path string) []byte {
    data, err := os.ReadFile(path)
    if err != nil {
        panic(err) // ❌ не используйте panic для IO ошибок
    }
    return data
}

// ✅ Хорошо
func ReadFile(path string) ([]byte, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, fmt.Errorf("чтение файла: %w", err)
    }
    return data, nil
}
```

### 4. Забытый `recover` в горутине

```go
// ❌ Плохо — паника убьёт всю программу
go func() {
    // код, который может паниковать
}()

// ✅ Хорошо
go func() {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("recovered: %v", r)
        }
    }()
    // код, который может паниковать
}()
```

## Заключение

Обработка ошибок в Go — это не просто проверка `if err != nil`. Это целая философия:

1. **Ошибки — это значения**, а не исключения
2. **Обрабатывайте ошибки явно** — не игнорируйте их
3. **Оборачивайте с контекстом** — используйте `%w` для сохранения цепочки
4. **Используйте `errors.Is` и `errors.As`** — для проверки обёрнутых ошибок
5. **Panic — только для фатальных ошибок** — не злоупотребляйте
6. **Логируйте в одном месте** — не дублируйте логи

Правильная обработка ошибок делает код надёжным, читаемым и простым в отладке.
