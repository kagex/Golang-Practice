# Пакеты (Packages) в Go: Организация, видимость и архитектура

## Что такое пакет и зачем он нужен

**Пакет (package)** — это фундаментальный механизм организации кода в Go. Пакет объединяет логически связанные файлы в единый модуль, определяя их область видимости и границы переиспользования.

Можно представить пакет как папку с файлами, решающими одну задачу:
* работа с базой данных (`database`);
* HTTP-обработчики (`handlers`);
* бизнес-логика (`service`);
* математические утилиты (`math`).

Без пакетов код превратился бы в монолитный файл на тысячи строк, который невозможно читать, тестировать или разрабатывать командой. Пакеты обеспечивают **инкапсуляцию** и **переиспользование**.

---

## Базовые правила

### 1. Объявление пакета
Первая строка любого `.go` файла должна содержать объявление пакета:
```go
package имя_пакета
```

⚠️ **Важно:** Пакет определяется именно этой директивой, а не именем папки. Хотя технически имя каталога и имя пакета могут отличаться, делать так **крайне не рекомендуется**. Большинство инструментов Go, разработчиков и IDE строго предполагают, что имя каталога совпадает с именем пакета.

```text
# ✅ Правильно и идиоматично:
calculator/
    package calculator

# ⚠️ Скомпилируется, но нарушает соглашения и запутывает инструменты:
calculator/
    package math 
```
Все файлы в одной директории **обязаны** принадлежать одному и тому же пакету. Компилятор выдаст ошибку, если в одной папке будут файлы с `package users` и `package admin`.

### 2. Пакет `main` и точка входа

Пакет с именем `main` — это такой же обычный пакет, как и любые другие. Однако он имеет особое значение для компилятора: он указывает, что этот код должен быть скомпилирован в исполняемый файл, а не в библиотеку.

Именно в пакете `main` должна находиться функция `func main()`, которая автоматически вызывается при запуске программы.

```go
package main

import "fmt"

func main() {
    fmt.Println("Программа запущена")
}
```

В одном модуле может находиться несколько пакетов `main` (например, в каталоге cmd/), каждый из которых компилируется в отдельный исполняемый файл.

## Документирование пакетов

В Go культура документирования встроена в язык. Документация пакета пишется в виде комментария, начинающегося со слов `// Package <имя>`, и располагается **непосредственно перед** директивой `package`.

```go
// Package calculator предоставляет функции
// для базовых математических вычислений.
package calculator
```
Этот комментарий будет автоматически подхвачен инструментами `go doc` и отображён на сайте `pkg.go.dev`. Для больших пакетов документацию часто выносят в отдельный файл `doc.go`.

## Стандартная библиотека

Go поставляется с богатой стандартной библиотекой. Вот наиболее часто используемые пакеты, которые не требуют установки:

* `fmt` — форматированный ввод-вывод.
* `strings` — манипуляции со строками.
* `strconv` — преобразование строк в числа и обратно.
* `time` — работа со временем и таймерами.
* `os` — функции операционной системы (файлы, переменные окружения).
* `net/http` — HTTP-клиент и сервер.
* `encoding/json` — кодирование и декодирование JSON.
* `sync` — примитивы синхронизации (`Mutex`, `RWMutex`, `WaitGroup`, `Once`).
* `context` — управление временем жизни операций, таймаутами, дедлайнами и отменой выполнения.
* `errors` — работа с ошибками.

## Импорт пакетов

Для использования кода из других пакетов применяется ключевое слово `import`.

### Группировка импортов
Хотя можно импортировать пакеты по одному, идиоматичный стиль Go (gofmt) предполагает группировку в блоке `import ()`:

```go
import (
    "fmt"
    "time"
    
    // Сторонние или собственные пакеты отделяются пустой строкой
    "github.com/gin-gonic/gin"
    "myproject/internal/config"
)
```

### Неиспользуемые импорты
Go-компилятор строго запрещает импортировать пакеты, которые не используются в коде. Это философия языка: код должен быть чистым, а зависимости — явными.
```go
import "fmt" // Ошибка: "fmt" imported and not used

func main() {}
```
💡 *Совет:* Современные IDE (VS Code с плагином Go, GoLand) удаляют неиспользуемые импорты автоматически при сохранении файла.

## Экспорт и область видимости

В Go **нет** ключевых слов `public`, `private` или `protected`. Видимость определяется исключительно **первой буквой имени** идентификатора (функции, переменной, константы, структуры или метода).

* 🟢 **Заглавная буква (Exported):** Идентификатор доступен для импорта из других пакетов.
* 🔴 **Строчная буква (Unexported):** Идентификатор виден *только* внутри текущего пакета.

```go
package calculator

// Экспортируется (виден снаружи как calculator.Add)
func Add(a, b int) int {
    return add(a, b) // Вызов внутренней функции
}

// Не экспортируется (виден только внутри пакета calculator)
func add(a, b int) int {
    return a + b
}

// Экспортируемая структура с неэкспортируемым полем
type User struct {
    Name string // Экспортируется
    age  int    // Не экспортируется, внешние пакеты не могут его читать или менять
}
```

## Специальные виды импорта

### 1. Псевдонимы (Aliases)
Полезно при конфликте имён или для сокращения длинных путей:
```go
import (
    myfmt "myproject/internal/customfmt"
    "fmt"
)

func main() {
    myfmt.Println("Custom")
    fmt.Println("Standard")
}
```

### 2. Импорт с подчёркиванием (Blank Import)
Используется **только** для выполнения побочных эффектов (side effects) пакета. Мы не используем функции пакета напрямую, но нам нужно, чтобы выполнился его код инициализации (например, регистрация драйвера БД).
```go
import _ "github.com/lib/pq" // Регистрирует драйвер "postgres" в database/sql
```

### 3. Точечный импорт (Dot Import)
```go
import . "fmt"
// Теперь можно писать просто Println("Hello") вместо fmt.Println
```
⚠️ **Настоятельно не рекомендуется** в продакшен-коде. Это ухудшает читаемость, так как невозможно понять, из какого пакета пришла функция. Допустим только в тестах (например, `import . "github.com/onsi/ginkgo/v2"`).

## Инициализация: функция `init()`

Каждый пакет может содержать одну или несколько функций `init()`.
* Они не принимают аргументов и ничего не возвращают.
* Они выполняются **строго один раз** для каждого пакета, даже если он импортирован в нескольких местах разными путями.

### Порядок выполнения
Инициализация в Go имеет строгий порядок. Переменные уровня пакета инициализируются **до** вызова `init()`.

```go
package main

import "fmt"

func load() int {
    fmt.Println("1. Загрузка данных")
    return 42
}

var x = load() // Выполняется первым

func init() {
    fmt.Println("2. Инициализация пакета (init)")
}

func main() {
    fmt.Println("3. Запуск программы (main)")
}
```
**Результат:**
```text
1. Загрузка данных
2. Инициализация пакета (init)
3. Запуск программы (main)
```

⚠️ **Почему `init()` часто считается антипаттерном:**  
Скрытая инициализация усложняет тестирование и отладку. Если `init()` паникует (например, не может подключиться к БД), программа падает до того, как вы успеете что-то перехватить.  
**Лучшая практика:** использовать явные конструкторы, например `func NewConfig() (*Config, error)`, и вызывать их вручную в `main()`.

## Go Modules и управление зависимостями

Начиная с Go 1.11, управление пакетами строится на **модулях (Go Modules)**. Модуль — это коллекция пакетов, версия которых управляется совместно.

Инициализация модуля:
```bash
go mod init github.com/username/myproject
```
Это создаёт файл `go.mod`:
```go
module github.com/username/myproject

go 1.22
```

### Синхронизация зависимостей (`go mod tidy`)
Просто написать `import "github.com/..."` в коде недостаточно. Чтобы Go скачал сторонний пакет и добавил его в `go.mod`, необходимо выполнить команду:

```bash
go mod tidy
```
**Что она делает:**
1. **Добавляет** отсутствующие зависимости, которые вы импортировали в коде.
2. **Удаляет** зависимости, которые больше не используются в коде.
3. **Обновляет файл `go.sum`**, записывая в него криптографические хеш-суммы скачанных модулей.

### Команда `go get`
`go get` — это основной инструмент для **осознанного добавления новых зависимостей** и **обновления** версий существующих модулей. Команда обновляет зависимости проекта и при необходимости загружает требуемые модули.

```bash
# Добавить/обновить зависимость до последней версии
go get github.com/gin-gonic/gin

# Указать конкретную версию или тег
go get github.com/gin-gonic/gin@v1.10.0

# Обновить все зависимости проекта
go get -u ./...
```

💡 **Важно:** Команда `go get` используется в первую очередь для управления зависимостями проекта, а не для установки исполняемых программ. Для установки CLI-инструментов (например, линтеров) следует использовать `go install`. После выполнения `go get` рекомендуется запускать `go mod tidy` для очистки.

💡 **Лайфхак для локальной разработки:**  
Если вы разрабатываете два модуля одновременно, используйте директиву `replace` в `go.mod`, чтобы не пушить изменения в Git:
```go
replace github.com/username/othermodule => ../path/to/local/othermodule
```

## Продвинутая архитектура: Пакет `internal`

В Go существует зарезервированное имя директории — **`internal`**.  
Любой пакет, находящийся внутри директории `internal` (или являющийся ею), **может быть импортирован только кодом, который находится в родительском дереве этого каталога**.

```text
myproject/
├── go.mod
├── main.go                 // ✅ Может импортировать internal/config
└── internal/
    └── config/
        └── config.go       // ❌ НЕ может быть импортирован извне myproject
```
Это идеальный механизм для сокрытия внутренней бизнес-логики от других модулей, даже если ваш код открыт как Open Source.

## Именование пакетов (Go Idioms)

Следуйте строгим правилам сообщества Go:
1. **Короткие и понятные имена:** `net`, `http`, `json`, `auth`.
2. **Одно слово, нижний регистр:** Никаких `camelCase`, `snake_case` или дефисов.
3. **Избегайте бессмысленных суффиксов:** Не называйте пакеты `utils`, `helpers`, `common` или `manager`. Называйте пакет по тому, *что он содержит* (например, `cache`, `validator`, `mailer`).
4. **Имя пакета должно совпадать с последним элементом пути импорта:** Если путь `github.com/user/project/payment`, пакет должен называться `package payment`.

## Полный пример: создание и использование собственного пакета

Давайте свяжем всё вместе в мини-проекте.

**Структура:**
```text
project/
├── go.mod
├── main.go
└── calculator/
    └── calculator.go
```

**1. `go.mod`**
```go
module github.com/username/project

go 1.22
```

**2. `calculator/calculator.go`**
```go
// Package calculator предоставляет математические функции.
package calculator

// Add экспортируется (заглавная буква)
func Add(a, b int) int {
    return a + b
}

// multiply не экспортируется (строчная буква)
func multiply(a, b int) int {
    return a * b
}
```

**3. `main.go`**
```go
package main

import (
    "fmt"
    "github.com/username/project/calculator" // Импорт по пути модуля
)

func main() {
    result := calculator.Add(2, 3) // ✅ Работает
    fmt.Println("Результат:", result)
    
    // fmt.Println(calculator.multiply(2, 3)) // ❌ Ошибка компиляции: неэкспортируемый идентификатор
}
```

## Типичные ошибки и как их избежать

| Ошибка | Почему это плохо | Решение |
| :--- | :--- | :--- |
| **Циклические зависимости** (`A` → `B` → `A`) | Компилятор не сможет определить порядок сборки. Ошибка: `import cycle not allowed`. | Вынесите общие типы в третий пакет (`domain`/`models`) или используйте интерфейсы для разрыва зависимости. |
| **Разные пакеты в одной папке** | Компилятор выдаст ошибку. | Один каталог = один пакет. |
| **Импорт и неиспользование** | Ошибка компиляции. | Удалите импорт или используйте `import _` для сайд-эффектов. |
| **Вызов `func lowerCase()` извне** | Ошибка компиляции (неэкспортируемый идентификатор). | Переименуйте в `UpperCase` или создайте функцию-обёртку. |
| **Игнорирование Go Modules** | Зависимости не фиксируются, сборка нестабильна. | Всегда используйте `go mod init`, `go get` и `go mod tidy`. |
| **Слишком большие пакеты** | Нарушение принципа единственной ответственности (SRP). | Дробите пакеты по функциональным границам. |

## Рекомендуемая структура проекта

Для средних и крупных приложений часто используется структура, близкая к [Standard Go Project Layout](https://github.com/golang-standards/project-layout). 

⚠️ *Важно:* Это популярная конвенция сообщества, а **не официальный стандарт языка**. Адаптируйте её под свои нужды. Использование каталога `pkg` необязательно; многие современные проекты вообще его не используют, предпочитая хранить весь код в `internal/` или корне модуля.

```text
myapp/
├── go.mod
├── go.sum
├── main.go                 # Точка входа, минимальная логика (wire-up)
│
├── cmd/                    # Дополнительные исполняемые файлы (если есть)
│   └── api/
│       └── main.go
│
├── internal/               # Приватный код, недоступный внешним модулям
│   ├── config/             # Чтение конфигов
│   ├── database/           # Подключение и миграции БД
│   ├── handlers/           # HTTP-обработчики (слой доставки)
│   ├── service/            # Бизнес-логика
│   └── models/             # Структуры данных (DTO, сущности БД)
│
└── pkg/                    # (Опционально) Публичные утилиты, которые можно переиспользовать в других проектах
    └── logger/
```

## Дополнительно: Директивы сборки (Build Constraints)

*Этот раздел относится к продвинутым возможностям языка.*

Пакеты и файлы могут компилироваться только при определённых условиях (ОС, архитектура, теги). Это управляется магическими комментариями в начале файла:

```go
//go:build linux && amd64

package mypackage
```
Или для разделения тестовых и продакшен-сборок:
```go
//go:build integration

package mypackage
```
Имя файла также может служить неявной директивой (например, `config_windows.go` скомпилируется только на Windows).

## Краткое резюме

1. Пакет — это директория с файлами, объявляющими одно и то же `package имя`.
2. Документируйте пакеты с помощью комментариев `// Package ...` перед директивой `package`.
3. Видимость определяется регистром первой буквы: `Name` (публично), `name` (приватно).
4. Используйте `internal` для сокрытия кода от внешних импортов.
5. Избегайте `init()`, отдавая предпочтение явным конструкторам `New...()`.
6. Именуйте пакеты коротко, в нижнем регистре, без суффиксов `utils`.
7. Для управления зависимостями используйте `go get` (для добавления) и `go mod tidy` (для очистки).
8. Грамотная архитектура пакетов предотвращает циклические зависимости и делает код тестируемым и масштабируемым.
