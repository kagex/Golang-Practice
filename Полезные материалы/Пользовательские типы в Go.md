# Пользовательские типы (Custom Types) в Go: От основ до глубин рантайма

## Введение
Go — язык со строгой статической типизацией. В отличие от динамических языков, где данные часто передаются как «сырые» словари или массивы, Go требует чёткого определения структур данных. 

Пользовательские типы позволяют:
1. **Добавить семантику:** `UserID` вместо `int64` сразу говорит о назначении данных.
2. **Обеспечить типобезопасность:** Компилятор не даст случайно передать `OrderID` туда, где ожидается `UserID`.
3. **Определить поведение:** Добавить методы к данным.

В Go есть три основных способа создания пользовательских типов: **определение на основе примитивов**, **структуры (structs)** и **интерфейсы (interfaces)**.

---

## 1. Пользовательские типы на основе примитивов

Самый простой способ создать свой тип — определить новый тип на основе базового (primitive).

```go
type UserID int64
type Status string
```

### Зачем это нужно?
**1. Типобезопасность (Type Safety)**
```go
func GetUser(id UserID) { ... }

var id int64 = 42
// GetUser(id) // ❌ Ошибка компиляции: cannot use id (variable of type int64) as UserID value
GetUser(UserID(id)) // ✅ Явное преобразование требуется
```

**2. Добавление методов**
К встроенным типам (`int`, `string`) нельзя добавить методы. К своим — можно.
```go
type Status string

const (
    StatusActive   Status = "active"
    StatusInactive Status = "inactive"
)

func (s Status) IsActive() bool {
    return s == StatusActive
}
```

> 💡 **Как часто спрашивают на собеседованиях:** В чем разница между `type MyInt int` и `type AliasInt = int`?
> *Ответ:* `type MyInt int` создает **новый** тип со своими методами, несовместимый с `int`. `type AliasInt = int` создает **алиас** (псевдоним), который полностью идентичен `int` и взаимозаменяем с ним. Алиасы нужны для рефакторинга и миграции кода.

## 2. Структуры (Structs)

Структуры — это составные типы, объединяющие несколько полей. Это фундамент моделирования данных в Go.

### Философия Zero Value
💡 **Правило Go:** *"Make the zero value useful"* (Сделай нулевое значение полезным). 
Структура без явной инициализации не должна приводить к панике или требовать обязательного вызова конструктора для базовой работы.
* `sync.Mutex` в нулевом состоянии уже готов к использованию.
* `bytes.Buffer` в нулевом состоянии пуст и готов к записи.
* `sync.WaitGroup` не требует инициализации.

```go
type User struct {
    ID     UserID
    Name   string
    Active bool
}

var u User // Zero value: {ID: 0, Name: "", Active: false}
```

### Композиция вместо наследования (Embedding)
В Go нет наследования классов. Вместо этого используется **встраивание (embedding)**.

💡 **Важно:** Embedding не создает отношение "является" (*is-a*), как наследование в ООП. `Dog` не *является* `Animal`. `Dog` *содержит* `Animal`. Именно поэтому композиция считается предпочтительным и более гибким способом повторного использования кода в Go.

```go
type Animal struct {
    Name string
}
func (a Animal) Speak() string { return "..." }

type Dog struct {
    Animal // Встроенное поле (анонимное)
    Breed  string
}

func (d Dog) Speak() string { return "Woof!" } // Перекрывает (shadows) метод Animal.Speak() и вызывается вместо него.

d := Dog{Animal: Animal{Name: "Rex"}, Breed: "Labrador"}
fmt.Println(d.Name)    // Rex (проксирование полей)
fmt.Println(d.Speak()) // Woof! (вызывается метод Dog, а не Animal)
```

### Сравнимые типы (Comparable Types)
Структуры можно сравнивать операторами `==` и `!=`, **только если все их поля являются сравнимыми**.
* ✅ **Можно сравнивать:** структуры, состоящие из примитивов, массивов (если их элементы сравнимы). Такие структуры можно использовать как ключи в `map`.
* ❌ **Нельзя сравнивать:** структуры, содержащие `slice`, `map` или `func`.

```go
type ComparableUser struct {
    ID   int
    Name string
}
a := ComparableUser{1, "Bob"}
b := ComparableUser{1, "Bob"}
fmt.Println(a == b) // ✅ true

// Можно использовать как ключ map:
m := make(map[ComparableUser]int) 

type NonComparableUser struct {
    Tags []string // slice не является сравнимым типом
}
// x := NonComparableUser{Tags: []string{"a"}}
// y := NonComparableUser{Tags: []string{"a"}}
// fmt.Println(x == y) // ❌ Ошибка компиляции: invalid operation: x == y

// Нельзя использовать как ключ map:
// var badMap map[NonComparableUser]int // ❌ Ошибка компиляции
```
> 💡 **Часто на собеседованиях спрашивают:** Почему структуры со слайсами нельзя сравнивать?
> *Ответ:* Слайс — это дескриптор (указатель, длина, емкость). Сравнение дескрипторов не гарантирует сравнение содержимого, а глубокое сравнение (deep equal) слишком дорого для оператора **==**. 
> Оператор **==** в Go всегда выполняет поверхностное сравнение. Для слайсов это означало бы неоднозначную семантику, поэтому они не являются сравнимыми типами.

## 3. Интерфейсы (Interfaces)

Интерфейсы определяют **поведение**, а не состояние. В Go они реализуются **неявно** (implicit implementation).

```go
type Writer interface {
    Write(p []byte) (n int, err error)
}

// MyFile не объявляет, что реализует Writer. 
// Но так как у него есть метод Write с нужной сигнатурой — он реализует интерфейс.
type MyFile struct { /* ... */ }
func (f *MyFile) Write(p []byte) (n int, err error) { /* ... */ }
```

💡 **Золотое правило Go:** Интерфейсы обычно объявляются **там, где они используются**, а не там, где они реализуются. Не создавайте интерфейсы «на будущее» в пакете реализации.

### Композиция интерфейсов
Интерфейсы можно встраивать друг в друга для создания более крупных абстракций:
```go
type ReadWriter interface {
    io.Reader
    io.Writer
}
```

### Type Assertions (Утверждения типа)
Чтобы извлечь конкретное значение из интерфейса `any` (или другого интерфейса), используется утверждение типа:
```go
var x any = "hello"

// Безопасное извлечение
if s, ok := x.(string); ok {
    fmt.Println("Это строка:", s)
}

// Type switch для множественной проверки
switch v := x.(type) {
case int:
    fmt.Println("Целое число:", v)
case string:
    fmt.Println("Строка:", v)
default:
    fmt.Println("Неизвестный тип")
}
```

## 4. Методы и приемники (Receivers)

Методы привязываются к типам через **приемники (receivers)**.

### Value Receiver vs Pointer Receiver
```go
type Counter struct { val int }

// Приемник-значение (работает с копией)
func (c Counter) GetVal() int { return c.val }

// Приемник-указатель (работает с оригиналом)
func (c *Counter) Increment() { c.val++ }
```

**Когда использовать указатель?**
1. Если метод **изменяет** состояние структуры.
2. Если структура **большая** (чтобы избежать дорогого копирования).
3. Если структура содержит поля, которые нельзя копировать (например, типы из `sync`).

💡 **Правило из Effective Go:** Если хотя бы один метод структуры использует pointer receiver, рекомендуется, чтобы **все** методы этой структуры использовали pointer receiver. Это обеспечивает согласованность API.

### Method Sets (Наборы методов)
Это одна из самых сложных тем для новичков. Набор методов типа определяет, какие интерфейсы он может реализовать.
* Набор методов типа `T` включает только методы с приемником `T`.
* Набор методов типа `*T` включает методы с приемником `T` **и** `*T`.

```go
type User struct{}
func (u User) A() {}
func (u *User) B() {}

var u User
var p *User = &u

// u.B() ✅ Работает! 
// Go автоматически делает (&u).B() как синтаксический сахар, 
// НО только если переменная u адресуема (addressable).

// User{}.B() // ❌ Ошибка: cannot call pointer method on User{} (литерал не адресуем)
```

**Влияние на интерфейсы:**
```go
type Runner interface {
    B()
}

var r Runner
// r = User{}   // ❌ Ошибка: User does not implement Runner (B method has pointer receiver)
r = &User{}     // ✅ Работает: *User реализует Runner
```
> 💡 **Как часто спрашивают на собеседованиях:** Почему `User` не реализует `Runner`, если у него есть метод `B()`?
> *Ответ:* Метод `B()` определен для указателя `*User`. Набор методов типа `User` содержит только `A()`. Интерфейс требует точного совпадения набора методов.

## 5. Как это устроено под капотом

### 5.1. Структуры в памяти: Padding и Alignment
Процессор читает память блоками (словами). Чтобы оптимизировать доступ, компилятор выравнивает поля структуры по их размеру (на 64-битных системах шаг выравнивания — 8 байт). Между полями добавляются пустые байты (**padding**).

```go
// ❌ Плохой порядок полей (тратится память)
type BadStruct struct {
    a bool  // 1 байт + 7 байт padding
    b int64 // 8 байт
    c bool  // 1 байт + 7 байт padding
} // Размер: 24 байта

// ✅ Хороший порядок полей (от больших к меньшим)
type GoodStruct struct {
    b int64 // 8 байт
    a bool  // 1 байт
    c bool  // 1 байт + 6 байт padding
} // Размер: 16 байт
```
💡 *Совет:* Для поиска неоптимального расположения полей можно использовать анализатор `fieldalignment` (из `golang.org/x/tools`) или настроить соответствующий линтер прямо в VS Code для подсветки ошибок в реальном времени.

### 5.2. Интерфейсы в памяти
Интерфейс в Go — это не просто абстракция. Упрощённо его можно представить как структуру из двух указателей.

**Для пустого интерфейса (`any` / `interface{}`):**
```go
type eface struct {
    _type *_type         // Указатель на описание типа в рантайме
    data  unsafe.Pointer // Указатель на сами данные
}
```

**Для непустого интерфейса (с методами):**
```go
type iface struct {
    tab  *itab          // Указатель на таблицу методов (interface table)
    data unsafe.Pointer // Указатель на данные
}
```
💡 *Нюанс:* Для маленьких значений (например, `int`, `bool`, указателей) рантайм может использовать **small value optimization** и хранить данные прямо внутри структуры `eface`/`iface` без дополнительной аллокации в куче.

`itab` кэшируется рантаймом. При первом присваивании значения интерфейсу Go ищет нужные методы и сохраняет указатели на них.

### 5.3. Escape Analysis
Компилятор Go решает, размещать переменную на стеке (быстро, очищается автоматически) или в куче (медленнее, требует работы GC).
Проверить это можно командой: `go build -gcflags="-m"`.

```go
// Случай 1: переменная убегает в кучу
func f() *User {
    u := User{Name: "Alice"}
    return &u // u escapes to heap: ссылка возвращается наружу
}

// Случай 2: переменная остаётся на стеке
func g() User {
    u := User{Name: "Bob"}
    return u // u does not escape: копируется значение
}
```
> 💡 **Как часто спрашивают на собеседованиях:** Всегда ли передача по указателю быстрее?
> *Ответ:* Нет. Для небольших структур передача по значению нередко эффективнее, поскольку не требует дополнительных аллокаций в куче и снижает нагрузку на сборщик мусора (GC).

## 6. Практические паттерны

### 1. Защита инвариантов (Unexported fields)
Если структура должна всегда находиться в валидном состоянии, скрывайте поля и используйте конструктор.
```go
type Config struct {
    timeout time.Duration // unexported
}

func NewConfig(timeout time.Duration) (*Config, error) {
    if timeout <= 0 {
        return nil, errors.New("timeout must be positive")
    }
    return &Config{timeout: timeout}, nil
}
```

### 2. Functional Options (Идиоматичный Go)
В Go для конфигурации сложных объектов чаще всего используется паттерн **Functional Options** (который опирается на замыкания):
```go
type Server struct { host string; port int; timeout time.Duration }

// Функция, изменяющая конфигурацию сервера.
// Такой подход позволяет легко добавлять новые параметры без изменения сигнатуры конструктора.
type Option func(*Server)

func WithPort(p int) Option           { return func(s *Server) { s.port = p } }
func WithTimeout(d time.Duration) Option { return func(s *Server) { s.timeout = d } }

func NewServer(host string, opts ...Option) *Server {
    s := &Server{host: host, port: 8080, timeout: 30 * time.Second} // дефолты
    for _, opt := range opts { opt(s) }
    return s
}
```

### 3. Generics (Обобщения)
Пользовательские типы могут быть параметризованы типами. Для операций, требующих сравнения, используется встроенное ограничение `comparable`, которое разрешает использование операторов `==` и `!=`.
```go
type Stack[T any] struct {
    items []T
}
func (s *Stack[T]) Push(item T) { s.items = append(s.items, item) }

func Contains[T comparable](slice []T, item T) bool {
    for _, v := range slice {
        if v == item { return true } // работает благодаря ограничению comparable
    }
    return false
}
```

## 7. Подводные камни

### 1. ⚠️ Nil интерфейс vs Типизированный nil
Интерфейс равен `nil` **только если и его тип, и его значение равны nil**.
```go
type MyError struct { msg string }
func (e *MyError) Error() string { return e.msg }

func check() error {
    var err *MyError = nil // Типизированный nil
    return err             // Возвращается как error (интерфейс)
}

err := check()
if err != nil {
    fmt.Println("Ошибка!") // ⚠️ Выведется! Хотя внутри nil.
}
```
> 💡 **Как часто спрашивают на собеседованиях:** Почему `err != nil` сработало, если мы присвоили `nil`?
> *Ответ:* Интерфейс `err` хранит пару `[тип: *MyError, значение: nil]`. Так как тип не nil, сам интерфейс не nil. Решение: возвращайте `error(nil)` явно.

### 2. ⚠️ Копирование типов из пакета `sync`
Типы синхронизации из пакета `sync` (например, `Mutex`, `RWMutex`, `WaitGroup`, `Cond`, `Once`) после первого использования **нельзя копировать**.

```go
type SafeCounter struct {
    mu sync.Mutex 
    val int
}
func (c SafeCounter) Get() int { // ❌ Приемник-значение копирует мьютекс!
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.val
}
```
**Решение:** Всегда используйте указатель `*SafeCounter`. `go vet`, который автоматически предупреждает о копировании таких типов при использовании value receiver.

### 3. ⚠️ Адресуемость элементов map
Элементы map в Go **не адресуемы**. Вы не можете изменить поле структуры, находящейся прямо в map, или взять его адрес.
```go
m := map[string]User{"a": {Name: "Bob"}}

// m["a"].Name = "Alice" // ❌ Ошибка компиляции: cannot assign to struct field m["a"].Name in map
// ptr := &m["a"]        // ❌ Ошибка компиляции: cannot take the address of map element

// ✅ Правильный способ:
u := m["a"]       // 1. Получаем копию
u.Name = "Alice"  // 2. Изменяем копию
m["a"] = u        // 3. Перезаписываем в map
```

### 4. ⚠️ Указатель на интерфейс
```go
func process(w *io.Writer) { ... } // ❌ Почти всегда ошибка
```
Интерфейс уже содержит указатель на данные. Указатель на интерфейс означает двойную косвенность и ломает проверку типов. Передавайте интерфейс по значению: `func process(w io.Writer)`.

## 8. Производительность

### Динамический диспетчеризация (Interface dispatch)
Вызов метода через интерфейс чуть медленнее прямого вызова, так как требует разыменования `itab`.
Разница обычно настолько мала, что выбор между интерфейсом и конкретным типом следует делать исходя из архитектуры программы, а не из микрооптимизаций.
Однако, начиная с современных версий, **компилятор Go способен в некоторых случаях девиртуализировать (devirtualize) и встроить (inline) вызовы интерфейсов**, если конкретный тип известен на этапе компиляции.

> ⚠️ **Важно:** Это оптимизация, а не гарантия. Не стоит рассчитывать на неё при проектировании API.

## 9. FAQ

**Почему нельзя взять адрес элемента map или его поля?**
Потому что map может быть перераспределена (rehashing) при добавлении новых элементов. Если бы Go разрешал брать адрес, этот указатель стал бы недействительным (dangling pointer). Go специально запрещает это на уровне компилятора ради безопасности памяти.

**Когда лучше использовать type alias (`=`) вместо нового типа?**
Используйте алиас, когда вам нужно временно переименовать тип при рефакторинге, или когда вы экспортируете внутренний тип пакета, не ломая обратную совместимость. Во всех остальных случаях для создания семантики используйте новый тип.

**Структура или Map? Что выбрать?**
* **Структура:** Типобезопасна, быстрее, занимает меньше памяти, поля известны на этапе компиляции.
* **Map:** Динамична, позволяет добавлять ключи в рантайме, но медленнее и требует приведения типов (если значение `any`).
* *Правило:* Используйте структуры для 95% задач. Map — только для словарей, кэшей и динамических наборов данных.

**Как сделать тип полностью неизменяемым (immutable)?**
1. Сделайте все поля unexported (с маленькой буквы).
2. Не пишите методов, изменяющих поля (только геттеры).
3. Если метод должен "изменить" объект, пусть он возвращает **копию** (как в `time.Time`).

## 10. Лучшие практики

> 💡 **Как часто спрашивают на собеседованиях:** Что значит "Accept interfaces, return structs"?
> *Ответ:* Функции должны принимать интерфейсы (для гибкости и тестирования), но возвращать конкретные структуры (чтобы caller мог использовать все методы, а не только интерфейсные). *Оговорка:* Это общая рекомендация, а не строгий закон (например, `sql.Open` возвращает `*sql.DB`).

1. **Держите интерфейсы маленькими.**
   Чем меньше методов в интерфейсе, тем легче его реализовать. Идеальный интерфейс — 1 метод (например, `io.Reader`, `fmt.Stringer`).
2. **Именуйте интерфейсы по поведению.**
   Обычно используется суффикс `-er`: `Reader`, `Writer`, `Handler`. Если интерфейс из одного метода `Write`, назовите его `Writer`.
3. **Следите за выравниванием и аллокациями.**
   Используйте анализатор `fieldalignment` и проверяйте escape analysis в hot-path участках кода.

## Заключение

Пользовательские типы в Go — это не просто способ сгруппировать данные. Это мощный инструмент для создания безопасной, выразительной и производительной архитектуры. 

* **Примитивные типы** дают семантику и типобезопасность.
* **Структуры** обеспечивают инкапсуляцию и композицию (а не наследование).
* **Интерфейсы** создают слабосвязанные абстракции, объявляемые там, где они нужны.

Понимание того, как эти типы размещаются в памяти (padding в структурах, `itab` в интерфейсах, escape analysis), отделяет просто "пишущего на Go" разработчика от инженера, способного оптимизировать высоконагруженные системы.

**Главное правило:**
Начинайте с простых структур и конкретных типов. Извлекайте интерфейсы только тогда, когда вам действительно нужна полиморфность или возможность подменить реализацию в тестах. 

> **Простота — одна из главных ценностей Go. Предпочитайте понятный и читаемый код сложным абстракциям**
